# Bucket Families — 学習用解像度バケットシステム

Bucket Families は、学習画像を**トークン数**（パッチ数）でグループ化する解像度バケットシステムです。各ファミリは複数のアスペクト比を含み、すべて同一のトークン数を生成するため、`torch.compile` は解像度ごとではなくファミリごとに1つのブロックグラフをトレースします。

## 画像とバケットのマッチング方法

マッチングは **面積 → ファミリ、次にアスペクト比 → バケット** の2段階で行われます。

### ステップ 1: 面積によるファミリマッチング

各ファミリには**標準ピクセル面積** = `tc × 256` があります（各 16×16 パッチが 256 ピクセルをカバーするため）。システムは、画像の実際の面積に最も近い標準面積を持つファミリを探します：

```
best_family = argmin |tc × 256 - image_width × image_height|
```

**例：** 800×800 の画像（面積 = 640,000）：

| ファミリ | トークン数 | 標準面積 | \|差分\| |
|----------|:----------:|:--------:|:--------:|
| S1 | 1024 | 262,144 | 377,856 |
| XS | 1680 | 430,080 | 209,920 |
| **S** | **2160** | **552,960** | **87,040 ← 最も近い** |
| M | 3600 | 921,600 | 281,600 |
| L | 4032 | 1,032,192 | 392,192 |
| S2 | 4096 | 1,048,576 | 408,576 |
| XL | 5040 | 1,290,240 | 650,240 |

→ 800×800 の画像は **S** ファミリ（tc=2160）に割り当てられます。

### ステップ 2: アスペクト比によるバケットマッチング

マッチングしたファミリ内で、画像のアスペクト比に最も近いバケットを探します：

```
best_bucket = argmin |bucket_AR - image_AR|    where AR = width / height
```

2つのバケットが同順位の場合（例：縦長と横長のミラー）、面積が最も近いものが選ばれます。

先ほどの例の続き — S ファミリには以下のメンバーがあります：

| バケット (W×H) | AR |
|----------------|-----|
| 384×1440 | 0.27 |
| 432×1280 | 0.34 |
| 480×1152 | 0.42 |
| 576×960 | 0.60 |
| 640×864 | 0.74 |
| 720×768 | 0.94 |
| (+ 横長ミラー) | (> 1.0) |

800×800 の画像の AR は 1.0 です。最も近いバケットは **720×768**（AR=0.94）、またはその横長ミラー **768×720**（AR=1.07）です。システムは AR が近い方を選択します — この場合は **720×768**（|0.94 - 1.0| = 0.06 対 |1.07 - 1.0| = 0.07）。

### リサイズとクロップ

バケットが選択された後：

1. **等方スケーリング** — 画像はバケット寸法を完全に覆うように（Lanczos 補間で）拡大縮小されます（レターボックスなし）。
2. **センタークロップ** — スケーリングされた画像はバケットの正確な (W, H) に中央クロップされます。

画像の面積が `tc × 256` に近い場合、クロップは最小限です。面積が標準面積から大きく離れている場合、より多くのクロップが発生します。

## 7つのファミリ

| ファミリ | トークン数 | 標準面積 (px) | メンバー解像度 (W×H) | 代表的な用途 |
|----------|:----------:|:-------------:|----------------------|-------------|
| **S1** | 1024 | 262,144 (0.26 MP) | 256×1024, 512×512 | 高速プロトタイピング、低VRAM学習 |
| **XS** | 1680 | 430,080 (0.43 MP) | 336×1280, 384×1120, 448×960, 480×896, 560×768, 640×672 | 軽量学習、中程度のARカバレッジ |
| **S** | 2160 | 552,960 (0.55 MP) | 384×1440, 432×1280, 480×1152, 576×960, 640×864, 720×768 | 速度と品質のバランス |
| **M** | 3600 | 921,600 (0.92 MP) | 480×1920, 576×1600, 640×1440, 720×1280, 768×1200, 800×1152, 960×960 | 高品質、良好なARカバレッジ |
| **L** | 4032 | 1,032,192 (1.03 MP) | 512×2016, 576×1792, 672×1536, 768×1344, 896×1152, 1008×1024 | デフォルト品質、高密度ARカバレッジ |
| **S2** | 4096 | 1,048,576 (1.05 MP) | 512×2048, 1024×1024 | 高品質の正方形および1:2 AR |
| **XL** | 5040 | 1,290,240 (1.29 MP) | 640×2016, 672×1920, 720×1792, 768×1680, 896×1440, 960×1344, 1008×1280, 1120×1152 | 最高品質 |

各ファミリには横長ミラー（W と H を入れ替えたもの）が自動的に含まれ、利用可能なアスペクト比が2倍になります（正方形バケットを除く）。

## トークン数とパフォーマンス

DiT は 16×16 パッチを使用するため、**トークン数 = (W ÷ 16) × (H ÷ 16)** となります。これは画像ごとにモデルが処理するパッチ数です。

### トークン数が重要な理由

`torch.compile` が有効な場合、コンパイラは異なるトークン数ごとに1つのブロックグラフをトレースします（`_native_flatten` 経由）。単一ファミリ内では、すべての解像度が1つのコンパイル済みグラフを共有します — アスペクト比に関係ありません。

**トークン数が低いほど：**

| 指標 | S1 (1024 トークン) | L (4032 トークン) | 比率 |
|------|:------------------:|:-----------------:|:----:|
| 画像あたりのピクセル数 | 262,144 | 1,032,192 | 1:3.9 |
| forward あたりのパッチ数 | 1,024 | 4,032 | 1:3.9 |
| バッチあたりのVRAM | 低い | 高い | — |
| ステップ/秒 | 高速 | 低速 | ~2–4× |
| 保持される視覚ディテール | 低い | 高い | — |

**トレードオフ：** 低トークン数ファミリ（S1、XS、S）は学習が大幅に高速でVRAM使用量も少なくなりますが、モデルが画像あたりに見るピクセル数が減るため、細かいディテールやテクスチャが失われます。高トークン数ファミリ（L、S2、XL）はより多くのディテールを保持しますが、より多くの計算リソースとVRAMが必要です。

### ゼロパディング保証

ファミリ内のすべてのバケットは、構造上トークン数を**正確に**満たします — バケット内でのパディングはありません。これにより以下が保証されます：

- Flash Attention がパディングマスクなしで実行される（アテンションリークなし）
- コンパイル済みグラフがイージャー（eager）forward パスとビット精度で一致する
- パディングトークンに対する無駄な計算がない

## マルチステージ学習戦略

速度と品質のバランスを取る実践的なアプローチとして、[Workflow エンジン](workflow.ja.md)を使った**2段階学習**があります：

### コンセプト

1. **ステージ 1 — 低解像度**（例：S1 または S ファミリ）：低トークン数でアダプタを学習します。モデルは計算コストのほんの一部で全体の構図、色、スタイルを学習します。
2. **ステージ 2 — 高解像度**（例：L または S2 ファミリ）：ステージ 1 のチェックポイントから高トークン数で学習を継続します。モデルはディテールとテクスチャを洗練させます。

### メリット

- 大規模データセットでの**総計算量の削減** — 高コストな高解像度ステージは数エポックだけで済みます。
- **速度とディテールのバランス** — 学習の大部分は低解像度で低コストに行われます。
- **クリエイティブな効果** — マルチ解像度学習が生み出すスタイライズされた見た目を好むユーザーもいます。

### リスク

- **品質低下** — 低解像度で学習された特徴が高解像度にうまく転用されない場合があり、アーティファクトや再現性の低下を引き起こす可能性があります。
- **過学習リスク** — ステージ 1 で低解像度にて過学習が発生した場合、ステージ 2 でそのアーティファクトが増幅される可能性があります。

### Workflow での設定方法

完全な手順は [Workflow ガイド — マルチステージの使用方法](workflow.ja.md#multi-stage-usage) を参照してください。概要：

1. **Preprocess** ステージを `bucket_families = "S1"` で追加 → 低解像度で画像を処理
2. **Train** ステージを `stop_epoch = 6` で追加 → 学習し、エポック 6 で停止
3. **Preprocess** ステージを `bucket_families = "L"` で追加 → 高解像度で画像を処理
4. **Train** ステージを追加 → ステージ 1 のチェックポイントから自動的に継続し、S1 と L の両方のキャッシュを使用

## CLI 使用方法

### 特定ファミリでリサイズ

```bash
python scripts/preprocess/resize_images.py \
  --src image_dataset/ \
  --dst post_image_dataset/resized/ \
  --bucket_families "L" \
  --tree
```

### 複数ファミリの指定

```bash
python scripts/preprocess/resize_images.py \
  --src image_dataset/ \
  --dst post_image_dataset/resized/ \
  --bucket_families "S,M,L" \
  --tree
```

### データセット分布分析

Workflow UI の Preprocess ステージにある「Analyze dataset」ボタンを使用すると、画像がファミリ間でどのように分布しているかを確認できます — すべてのファミリが利用可能な場合（自然分布）と、選択したファミリのみの場合の両方を確認できます。
