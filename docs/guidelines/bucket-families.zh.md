# Bucket Families —— 训练用的分辨率分桶系统

Bucket Families 是一个分辨率分桶系统，按 **token 数量**（patch 数）对训练图像进行分组。每个家族包含多个宽高比，它们都产生完全相同的 token 数量，因此 `torch.compile` 会为每个家族而非每个分辨率追踪一个 block 计算图。

## 图像如何匹配到分桶

匹配过程分两步：**面积 → 家族，然后宽高比 → 分桶**。

### 第一步：面积匹配到家族

每个家族都有一个 **标准像素面积** = `tc × 256`（因为每个 16×16 的 patch 覆盖 256 个像素）。系统找到标准面积与图像实际面积最接近的家族：

```
best_family = argmin |tc × 256 - image_width × image_height|
```

**示例：** 一张 800×800 的图像（面积 = 640,000）：

| 家族 | Token 数量 | 标准面积 | \|差值\| |
|------|:----------:|:--------:|:-------:|
| S1 | 1024 | 262,144 | 377,856 |
| XS | 1680 | 430,080 | 209,920 |
| **S** | **2160** | **552,960** | **87,040 ← 最近** |
| M | 3600 | 921,600 | 281,600 |
| L | 4032 | 1,032,192 | 392,192 |
| S2 | 4096 | 1,048,576 | 408,576 |
| XL | 5040 | 1,290,240 | 650,240 |

→ 这张 800×800 的图像被分配到 **S** 家族（tc=2160）。

### 第二步：宽高比匹配到分桶

在匹配到的家族内，系统找到宽高比与图像最接近的分桶：

```
best_bucket = argmin |bucket_AR - image_AR|    其中 AR = width / height
```

如果两个分桶并列（例如竖版和横版的镜像），则选择面积更接近的那个。

继续上面的示例 —— S 家族有以下成员：

| 分桶 (W×H) | AR |
|-------------|-----|
| 384×1440 | 0.27 |
| 432×1280 | 0.34 |
| 480×1152 | 0.42 |
| 576×960 | 0.60 |
| 640×864 | 0.74 |
| 720×768 | 0.94 |
| (+ 横版镜像) | (> 1.0) |

一张 800×800 的图像 AR = 1.0。最接近的分桶是 **720×768**（AR=0.94）或其横版镜像 **768×720**（AR=1.07）。系统选择 AR 更接近的那个 —— 本例中是 **720×768**（|0.94 - 1.0| = 0.06 vs |1.07 - 1.0| = 0.07）。

### 缩放与裁剪

选定分桶后：

1. **各向同性缩放** —— 使用 Lanczos 插值对图像进行缩放，使其完全覆盖分桶尺寸（不会出现黑边）。
2. **中心裁剪** —— 将缩放后的图像从中心裁剪到分桶的精确 (W, H)。

当图像面积接近 `tc × 256` 时，裁剪量很小。当面积远离标准面积时，裁剪量会更多。

## 七个家族

| 家族 | Token 数量 | 标准面积 (px) | 成员分辨率 (W×H) | 典型用途 |
|------|:----------:|:-------------:|------------------|----------|
| **S1** | 1024 | 262,144 (0.26 MP) | 256×1024, 512×512 | 快速原型验证，低显存训练 |
| **XS** | 1680 | 430,080 (0.43 MP) | 336×1280, 384×1120, 448×960, 480×896, 560×768, 640×672 | 轻量训练，适中的宽高比覆盖 |
| **S** | 2160 | 552,960 (0.55 MP) | 384×1440, 432×1280, 480×1152, 576×960, 640×864, 720×768 | 速度与质量的平衡 |
| **M** | 3600 | 921,600 (0.92 MP) | 480×1920, 576×1600, 640×1440, 720×1280, 768×1200, 800×1152, 960×960 | 高质量，良好的宽高比覆盖 |
| **L** | 4032 | 1,032,192 (1.03 MP) | 512×2016, 576×1792, 672×1536, 768×1344, 896×1152, 1008×1024 | 默认质量，密集的宽高比覆盖 |
| **S2** | 4096 | 1,048,576 (1.05 MP) | 512×2048, 1024×1024 | 高质量的正方形和 1:2 宽高比 |
| **XL** | 5040 | 1,290,240 (1.29 MP) | 640×2016, 672×1920, 720×1792, 768×1680, 896×1440, 960×1344, 1008×1280, 1120×1152 | 最高质量 |

每个家族自动包含横版镜像（W 和 H 互换），将可用的宽高比翻倍（正方形分桶除外）。

## Token 数量与性能

DiT 使用 16×16 的 patch，因此 **token 数量 = (W ÷ 16) × (H ÷ 16)**。这是模型处理每张图像时的 patch 数量。

### 为什么 token 数量很重要

当启用 `torch.compile` 时，编译器会为每个不同的 token 数量追踪一个 block 计算图（通过 `_native_flatten`）。在同一个家族内，所有分辨率共享一个编译后的计算图 —— 无论宽高比如何。

**较低的 token 数量意味着：**

| 指标 | S1 (1024 tokens) | L (4032 tokens) | 比率 |
|------|:-----------------:|:----------------:|:----:|
| 每张图像的像素数 | 262,144 | 1,032,192 | 1:3.9 |
| 每次前向的 patch 数 | 1,024 | 4,032 | 1:3.9 |
| 每批次显存占用 | 较低 | 较高 | — |
| 每秒步数 | 更快 | 更慢 | ~2–4× |
| 保留的视觉细节 | 较少 | 较多 | — |

**权衡：** 低 token 数量的家族（S1、XS、S）训练速度显著更快且显存占用更少，但模型看到的每张图像的像素更少 —— 精细细节和纹理会丢失。高 token 数量的家族（L、S2、XL）保留更多细节，但需要更多的计算资源和显存。

### 零填充保证

家族中的每个分桶在构造上 **精确** 填满其 token 数量 —— 不存在分桶内的填充。这意味着：

- Flash Attention 无需填充掩码即可运行（无注意力泄漏）
- 编译后的计算图与 eager 前向路径位级精确一致
- 没有在填充 token 上浪费计算

## 多阶段训练策略

一种平衡速度与质量的实用方法是使用 [Workflow 引擎](workflow.zh.md)进行 **两阶段训练**：

### 概念

1. **阶段 1 —— 低分辨率**（例如 S1 或 S 家族）：在低 token 数量下训练适配器。模型以一小部分的计算成本学习整体构图、色彩和风格。
2. **阶段 2 —— 高分辨率**（例如 L 或 S2 家族）：从阶段 1 的检查点继续训练，使用高 token 数量。模型细化细节和纹理。

### 优势

- 对于大型数据集 **减少总计算量** —— 昂贵的高分辨率阶段只需少量 epoch。
- **速度与细节的平衡** —— 大部分学习在低分辨率下低成本完成。
- **创意效果** —— 一些用户偏好多分辨率训练产生的风格化效果。

### 风险

- **质量退化** —— 在低分辨率下学到的特征可能无法很好地迁移到高分辨率，可能导致伪影或保真度降低。
- **过拟合风险** —— 如果阶段 1 在低分辨率下过拟合，阶段 2 可能会放大这些伪影。

### 如何在 Workflow 中设置

参见 [Workflow 指南 —— 多阶段用法](workflow.zh.md#多阶段用法)获取完整说明。简要步骤如下：

1. 添加 **Preprocess** 阶段，设置 `bucket_families = "S1"` → 以低分辨率处理图像
2. 添加 **Train** 阶段，设置 `stop_epoch = 6` → 训练并在第 6 个 epoch 停止
3. 添加 **Preprocess** 阶段，设置 `bucket_families = "L"` → 以高分辨率处理图像
4. 添加 **Train** 阶段 → 自动从阶段 1 的检查点继续，同时使用 S1 和 L 缓存

## CLI 用法

### 使用指定家族进行缩放

```bash
python scripts/preprocess/resize_images.py \
  --src image_dataset/ \
  --dst post_image_dataset/resized/ \
  --bucket_families "L" \
  --tree
```

### 使用多个家族

```bash
python scripts/preprocess/resize_images.py \
  --src image_dataset/ \
  --dst post_image_dataset/resized/ \
  --bucket_families "S,M,L" \
  --tree
```

### 数据集分布分析

使用 Workflow UI 中 Preprocess 阶段的"分析数据集"按钮，可以查看图像在各家族间的分布情况 —— 包括启用所有家族时的自然分布，以及仅使用所选家族时的分布。
